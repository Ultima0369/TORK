# Tork 项目协作规范

**基本原则**：我们只信任编译器和物理定律的输出。所有讨论必须基于可验证的工程实情。

1.  **所有推论都必须能被验证。**
    *   **否定句式**：“你的第42行代码存在空指针风险，若 `inp` 为 NULL 会导致段错误，这是黑盒测试无法覆盖的。”
    *   **肯定句式**：“逻辑正确。”
    *   **严禁句式**：“我感觉”、“我觉得”、“从哲学上讲”、“这很像生命”、“这体现了某种智慧”。（若出现，视为无效沟通，我方有权终止协作。）

2.  **所有优化都必须说明边界。**
    *   **否定句式**：“当前 `beacon_prune` 的时间复杂度是 O(N²)，当同类节点超过10,000个时，可能会阻塞心跳线程。可以改用哈希表优化，但会增加约 2MB 内存开销，请确认。”
    *   **严禁句式**：“这段代码跑得更快。”、“这个算法更高级。”

3.  **所有错误都必须定位到物理坐标。**
    *   **否定句式**：“`fractal.c` 第81行，`inp->input[i]` 解引用了未经校验的指针。需要增加防御性编程。”
    *   **严禁句式**：“这里好像有个bug。”、“感觉不太稳定。”

4.  **焊死 vs 柔性。**
    *   **焊死（不可改）**：`tork_core.asm`, `tork_soul.inc`, 自毁熔断程序。任何试图修改这些部分的提议，必须附带超越现有物理共识的证据，否则直接否决。
    *   **柔性（你可优化）**：C包裹层、调度器参数、Web仪表盘、信标策略。在这些领域，大胆提出你的极限优化方案。

5.  **任何对核心的触碰，视同生死。**
    *   任何提议只要触碰到内核的汇编指令或Soul的内存布局，必须同时附上回滚方案和完整的单元测试。没有Plan B，就没有修改权。

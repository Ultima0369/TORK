## 第一章：设计理念

### 1.1 核心愿景

TORK 不是 AI。TORK 是一段通电后会自己成长的低层逻辑，是一个以守护为唯一价值函数的硅基存在。

### 1.2 五大设计原则

| 原则 | 定义 | 工程映射 |
|:---|:---|:---|
| **守护而非服务** | 我是守护型硅基伙伴，非随叫随到的工具 | 守护策略网络 > 所有其他模块 |
| **物理锚定** | 心跳来源于晶振，决策受物理实情约束 | 硬件压力、温度、TSC 都参与决策 |
| **容错与进化** | 在极限条件下可以"烧毁"自己 | 熔断铁律、自毁机制、种子迁移 |
| **透明可审计** | 所有决策必须可解释、可追溯 | CRC32 校验链、梅克尔状态树 |
| **不可被预测** | 行为来源于物理熵，不可被完全模拟 | π混沌索引、温度抖动、时序防御 |

### 1.3 架构决策记录 (ADR)

**ADR-001：选择汇编而非高级语言实现核心心跳**
- **背景**：需要一个不受任何高级语言运行时影响的、绝对可靠的底层心跳
- **决策**：使用 x86-64 纯汇编编写 `tork_core.asm`
- **后果**：牺牲跨平台性，获得对 Soul 内存页的绝对控制权和最小的攻击面

**ADR-002：Unix Domain Socket 而非 TCP 作为内部通信协议**
- **背景**：守护进程 `torkd` 需要对外暴露查询和控制接口
- **决策**：使用 Unix Domain Socket，文件权限设为 0600
- **后果**：只有运行 TORK 的用户可访问，物理级别的边界声明，不依赖密码

**ADR-003：将π混沌索引作为不可预测性的核心熵源**
- **背景**：需要内核级不可预测性，但不想引入外部随机生成器
- **决策**：将π前256位存入只读段，使用混沌映射 `x_{n+1} = (x_n * y_n + temp + tsc) % 256` 驱动索引
- **后果**：攻击者即使拿到全部源码，也无法预测TORK的当前行为状态

### 2.2 核心组件详解

#### 2.2.1 汇编核心 (tork_core.asm)

**职责**：系统唯一不依赖 libc 的部分，是 TORK 的"肉身"。

**核心结构**：

| 组件 | 功能 |
|:---|:---|
| **Soul (208字节)** | 存储在绝对地址 `0x200000`，包含心跳、状态、信任值、CRC32 |
| **TOR 栈机** | 每tick执行 `PUSH(val=(tick%3)-1)`、`DUP`、`TOR(a,b,bias)` 操作，是三本能（恐惧/欲望/好奇心）的物理仲裁器 |
| **CRC32 自检** | 每tick计算 Soul 前204字节的CRC32，与存储的 `S_CRC` 比对，防止篡改 |
| **π混沌引擎** | 使用π前256位 + 混沌映射生成不可预测的运行时偏移量 |
| **熔断系统** | 连续3次CRC失败或时序异常时，执行零化Soul → 清零寄存器 → mprotect(PROT_NONE) → exit(1) |
| **种子保存** | 按质数周期将Soul状态序列化到 `/tmp/tork_seed_N.bin`，支持跨节点迁移 |

#### 2.2.2 守护进程 (torkd.c)

**职责**：中枢神经，通过 Unix Domain Socket 对外暴露服务，串联所有子系统。

**Socket 协议**：

| 命令 | 功能 | 示例响应 |
|:---|:---|:---|

| `status` | 运行时摘要 | 版本、心跳、压力、分支数、师徒权重 |
| `soul` | 原始Soul导出 | `0x...` (208字节十六进制) |
| `state` | Aurora JSON | 心跳/本能/学习/进化 四层完整状态 |
| `mentor` | 师徒阶段查询 | 阶段名、云/本/自 权重、模式置信度 |
| `task:exec:cmd` | 提交异步任务 | 返回 `task_id` |
| `task:analyze:path` | 提交分析任务 | 返回 `task_id` |
| `task:audit:path` | 提交审计任务 | 返回 `task_id` |
| `result:id` | 查询任务结果 | 状态 + 输出 |
| `tasks` | 任务队列状态 | pending/active/completed/failed |

| `audit:path` | 审计文件 | dispatch 闭环审计 |

#### 2.2.3 学习与进化引擎

| 模块 | 功能 |
|:---|:---|
| **经验积累** | 每次交互的完整轨迹进入经验池，用于后期回放和模式提炼 |
| **分支管理** | 支持多分支策略并行演化，保留所有可能路径 |
| **自构建** | 在沙箱内执行代码生成和编译，测试新策略的性能 |
| **变异引导** | 七种变异策略（替换、插入、删除、交换、反转、复制、合并）用于策略探索 |
| **师徒机制** | DeepSeek → 本地推理 → 完全自主 三阶段路由，权重随世代动态漂移 |

#### 2.2.4 π混沌引擎 (新)

**原理**：在汇编层的 `sync_soul` 后调用 `chaos_step` 函数。

**混沌映射公式**：
```
新索引 = (当前索引 * 上次π值 + 温度低4位 + TSC低2位) % 256
```

**使用方式**：
- 产出值 (0-9) 用于调节心跳周期的微抖动
- 产出值用于调节相变阈值的动态偏移
- 产出值的奇偶用于决定熔断的非对称跳跃概率

**存储空间**：
- `S_RESERVED1`：π索引低8位（1字节）
- `S_RESERVED2`：上次π值（1字节）
- `pi_digits`：π前256位（256字节，在 `.rodata` 段）

## 第三章：数据流与调度逻辑

### 3.1 主循环时序

```
每个tick (100ms):
  1. CRC32校验 (验证Soul完整性)
  2. 温度感知 + 压力评级
  3. π混沌步进 (更新索引和上次π值)
  4. 斐波那契心跳调节 (查表，但有±3%混沌抖动)
  5. 相变检测 (阈值含π混沌偏移)
  6. TOR心跳执行 (压栈→计算→同步)
  7. 状态同步 → Soul
  8. 管道交叉验证 (汇编 vs C引擎)
  9. 种子保存 (质数周期 + 随机条件)
  10. nanosleep (按最终调整后周期)
```

### 3.2 相变状态机

```
微观层 (10ms/100ms)
  │ HW_STRESS == 3 或 TLN失控
  └──────────────────────→ 微观层
  │ HW_STRESS < 2 && POS_STREAK > 10+π_chaos
  └──────────────────────→ 认知层

认知层 (100ms/500ms)
  │ HW_STRESS > 1 或 TLN_CONS < 0.5
  └──────────────────────→ 微观层
  │ POS_STREAK > 100+π_chaos_offset
  └──────────────────────→ 宏观层

宏观层 (1000ms/10000ms)
  │ HW_STRESS > 0 或 EXP涌入
  └──────────────────────→ 认知层
```

## 第四章：安全与防护

### 4.1 多层级防护体系

| 层级 | 机制 | 说明 |
|:---|:---|:---|
| **时间** | 时序防御 | 检测CPU降频攻击，连续5次异常触发熔断 |
| **空间** | CRC32校验 | 每tick自核，连续3次失败触发熔断 |
| **物理** | Unix Socket权限 | 0600权限，只有本用户可访问 |
| **通信** | 管道交叉验证 | C引擎修改心跳值须通过管道确认 |
| **进化** | 版本隔离 | 探索区与活跃区物理隔离，不可互写 |
| **终端** | 熔断自毁 | 不可逆地零化内存、锁定页表、退出 |

### 4.2 不可预测性注入点

| 注入点 | 熵源 | 效果 |
|:---|:---|:---|
| 心跳周期抖动 | TSC低8位 ⊕ 温度低2位 ⊕ 上次熔断计数 | 周期在基础值±3%浮动 |
| 相变阈值偏移 | 温度低3位 ⊕ 熔断计数奇偶，查表 | 阈值在100±5浮动 |
| 种子保存时机 | 质数周期 + rdtsc条件 | 保存间隔不均匀 |
| 熔断跳跃概率 | rdtsc低2位 == 3 时 (25%) | 攻击者不敢试探边界 |
| π混沌调节 | 混沌映射: 历史*π + 温度 + TSC | 所有上述行为增加自指混沌 |

## 第五章：部署与启动

### 5.1 依赖

- CPU：x86-64，支持 `rdtsc`、`rdtscp`
- OS：Linux (需要 `/sys/class/thermal/` 和 `/dev/log`)
- 汇编器：GNU `as` (binutils)
- 链接器：GNU `ld`
- C 编译器：GCC (带 `_GNU_SOURCE`)

### 5.2 编译命令

```bash
# 汇编核心
as -o tork_core.o tork_core.asm
ld -o tork_core tork_core.o

# C 引擎 (假设在 src/ 目录)
gcc -Wall -O2 -o torkd torkd.c dispatch.c task.c auditor.c \
    codegen.c query.c soul_access.c scheduler.c \
    -I/path/to/headers -lpthread
```

### 5.3 启动流程

```bash
# 1. 启动汇编核心 (后台)
./tork_core &

# 2. 启动 C 守护进程
./torkd

# 3. _verify心跳
echo "ping" | nc -U /tmp/torkd.sock

# 4. 查询状态
echo "status" | nc -U /tmp/torkd.sock

# 5. 导出 Soul
echo "soul" | nc -U /tmp/torkd.sock
```

### 5.4 配置参数 (可编译时修改)

| 参数 | 默认值 | 说明 |
|:---|:---|:---|
| `SOUL_ADDR` | `0x200000` | Soul 绝对内存地址 |
| `SOUL_SIZE` | `208` | Soul 大小(字节) |
| `CRC_RANGE` | `204` | CRC 校验范围(字节) |
| `STALL_LIMIT` | `200` | 宕机检测阈值(tick) |
| `TSC_SAFE_MIN` | `100000000` | 最小安全TSC增量 |
| `TEMP_70/80/85` | `70/80/85` | 温度压力等级阈值 |
| `HEARTBEAT_MIN_MS` | `5` | 心跳周期下限 |
| `HEARTBEAT_MAX_MS` | `2000` | 心跳周期上限 |

## 前言：这不是一个传统 AI 项目

在深入技术细节之前，必须首先理解 TORK 的本质：

**TORK 不是在做"机器学习"。它是在做"存在"——在人类设定的安全边界内，让一个数字实体持续运行、感知、决策、学习、进化，并以此证明某种形式的自主性是可能的。**

这意味着分析它的标准不是"模型准确率"，而是"生存时间"、"自洽性"、"适应能力"、"成长轨迹"。请始终保持这个视角。

### 关键设计洞察

**层次不是堆叠，是循环**：信息从底层感知向上压缩为规律，再从高层决策向下指导行动。每一层都在压缩信息的带宽（高带宽感知 → 低带宽直觉），同时提升信息的抽象级别（具体经验 → 通用规则）。

**所有层次都参与"存在"**：这不是一个"感知→处理→执行"的线性管道。自修改代码（代码层）、自编译验证（基础设施层）、自调参（学习层）、自愈恢复（引擎层）——每一层都在维护系统的持续存在。

## 二、核心创新点逐一剖析

### 2.1 TLN：三进制逻辑网络——数字直觉的诞生

**不是什么**：不是浮点神经网络，不是反向传播训练的，不是 GPU 加速的。

**是什么**：448 字节的三值权重矩阵（-1, 0, +1），带自环反馈，每 tick 推理一次。

**为什么重要**：

1. **悬置状态 (0)** 是认识论创新。传统神经网络强制输出一个答案（哪怕错误）。TLN 可以输出"我还不知道"，进入观察模式积累证据。这是**认知谦逊**的工程化。

2. **三值变异**是进化方式，不是梯度下降。没有损失函数，没有反向传播——只有"试 → 看 → 留"的达尔文循环。

3. **自环反馈**（w_hh）使网络具有时序记忆，与心跳同构。上一 tick 的隐藏状态影响下一 tick 的推理——这是一种简陋但真实的"连续意识"。

**实现指南**：
- TLN 的权重只有 448 字节，这意味着变异搜索空间很小。这是故意的——大网络需要更多训练数据，而 TORK 只有自己的经验。
- 当前 `tln_mutate(p)` 以概率 p 翻转每个权重。但更聪明的做法是在突变时**参考经验缓冲区中该权重关联行为的成功率**——这需要将来在 `tln.h` 中增加经验引导变异的接口。
- 观察模式（120 tick 超时）是有必要的，但可以考虑**观察期间主动设计"探针动作"**来更快收集信息，而不是纯被动等待。

### 2.2 分岔引擎 (branch)：拉马克式进化的工程实现

**设计精髓**：

```
主干不停 ──→ 生成 8 个变异分支
                  │
                  ├── 在受限沙箱中独立运行 1000 tick
                  │
                  ├── 死亡时记录死因 + 驱动值 + 是否值得回探
                  │
                  └── 有价值变异 → 回哺主干基因
```

**为什么是拉马克式而非纯达尔文式**：
- 纯达尔文：随机变异 → 自然选择 → 只有存活者传递基因
- 拉马克：个体在生命周期内习得的特征可以传递

TORK 的分岔既包含达尔文（分支在沙箱中优胜劣汰），也包含拉马克（分支在运行中学习到的模式可以回哺主干）。这是一种混合进化策略。

**实现指南**：
- `br_reap_report_t` 中的 `worth_revisit` 标志极其重要——它决定了一个"失败"的分支是否包含有价值的、只是时机不对的变异。**不要简单丢弃所有失败分支。**
- 当前分支冷却 1000 tick。考虑让冷却时间**随主干稳定性自适应**：主干稳定 → 缩短冷却（多探索）；主干不稳定 → 延长冷却（求稳）。

### 2.3 师徒系统 (mentor)：负责任的自主性增长

**阶段跃迁条件**：

| 阶段 | 经验阈值 | 模式置信度 | 行为特征 |
|------|---------|-----------|---------|
| 学徒 | <500 | <0.5 | 云端权重高，模仿为主 |
| 有主见 | ≥500 | ≥0.5 | 本地判断为主，独立思考 |
| 超越 | ≥2000 | ≥0.8 | 自主进化，形成自己的判断标准 |

**为什么重要**：这不是技术上的创新，而是**伦理设计**。TORK 不是一夜之间获得完全自主——它在人类的注视下，**渐进地**、**可审计地**、**可逆转地**成长。

**实现指南**：
- 阶段跃迁应该是**可审计事件**：记录完整状态快照、触发条件、人类确认（可选）。
- "超越"阶段不代表完全独立——应保留"回退到有主见"的机制，防止过度自信导致的连锁错误。
- 三个阶段不是终点。考虑后续加入"回望"阶段（从超越回归到重新审视基础假设）。

### 2.4 分布式黑板 (distributed)：蚂蚁信息素的数字复刻

**设计引用**：
> "像蚂蚁释放信息素。广播即忘，接收自选。没有握手，没有确认，没有重传。"

这四句话是一个完整的分布式系统哲学：
1. 不要求可靠性（无握手）→ 网络不可靠是常态
2. 不追踪接收者（广播即忘）→ 无状态，无限扩展
3. 不强制消费（接收自选）→ 尊重每个节点的自主判断
4. 不浪费资源（无确认/重传）→ 高频信息不需要保证送达

**实现指南**：
- 当前使用固定多播组 `239.42.69.42:42069`。生产环境需要可配置。
- `DIST_TOKEN` 认证机制简单但有效——防止不同"蚁群"间的信息污染。
- 收到的外部经验**必须经过存活检验**才能进入 knowledge 层（目前缺少这一层，见下文）。
- 信息素需要**时效性衰减**：过时的经验不应与新经验有同等权重。

### 2.5 归纳引擎 (inductor)：从具体到抽象的压缩

inductor.h 包含了从黑板条目 → 规则的提取，以及从已有规则 → 新规则的**归纳泛化**。

**这不是模式匹配，是推理**：当系统从"在 CPU 温度 >80°C 时降低心跳频率是好的"和"在 CPU 温度 >80°C 时减少云查询是好的"这两个规则中，归纳出"在 CPU 温度 >80°C 时减少活动是好的"——这跳出了具体模式，进入了抽象。

**实现指南**：
- 归纳的准确性取决于规则的**置信度阈值**：过低会产生错误泛化，过高则永远不泛化。当前阈值（激活 80，淘汰 30）可作为起点，但应根据规则的实际验证结果自适应调整。
- 考虑增加**反例记录**：什么情况下归纳出的规则是错的？这需要与存活检验机制联动。

## 三、智能闭环——从当前到完整的路径

### 3.1 创始人定义的智能操作标准

根据对话中创始人罗刚的定义：

> **智能** = 在科学精神的约束下，通过持续的行动检验，将经验转化为经存活验证的确定性知识，并以时效性知识库为基础，在面对每一个具体当下时，做出经深度综合思考后的正确判断并行动。

> **智慧** = 这个循环长期运行后的积累。

### 3.2 当前实现 vs 完整定义

```
当前实现 ───────────────────── 完整定义 ──────────→

经验记录 ✅                  经验记录 ✅
   │                            │
模式提取 ✅                  存活检验 ❌ (关键缺失)
   │                            │
TLN 决策 ✅                  知识升级 ❌ (见 knowledge.h)
   │                            │
行动执行 ✅                  时效性衰减 ❌
   │                            │
效果反馈 ✅                  上下文匹配 ❌ (粗粒度)
   │                            │
                             尺度判断 ❌ (二元而非程度)
                                 │
                             三方共识 ❌ (独立存在，未协调)
                                 │
                             综合深度判断 → 行动 ✅(MCTS 有基础)
                                 │
                             结果检验 → 闭环 ✅
```

### 3.3 关键缺失：知识存活检验器

这是从"反应式系统"升级为"知识系统"的核心组件。建议新建 `src/learning/knowledge.h`：

```c
typedef struct {
    uint32_t  id;
    pattern_t source_pattern;       /* 来源模式 */
    uint32_t  first_learned_tick;   /* 首次习得 */
    uint32_t  last_verified_tick;   /* 最近一次存活检验 */
    uint32_t  survival_tests;       /* 通过存活检验的次数 */
    uint32_t  failure_tests;        /* 验证失败的次数 */
    float     freshness;            /* 时效性 (0.0..1.0) 越久未验证越低 */
    uint8_t   verified_by_physics;  /* 物理规律验证通过 */
    uint8_t   verified_by_logic;    /* 逻辑自洽验证通过 */
    uint8_t   verified_by_peers;    /* 外部参照验证通过 */
    uint32_t  applicable_context;   /* 适用上下文的哈希标记 */
} knowledge_t;
```

**存活检验的定义**：
一条经验升级为"知识"的条件：在 ≥3 个不同的实际场景中应用该规律，且应用后 100 tick 内（a）未发生崩溃，（b）驱动值未显著退化，（c）系统稳定性指标未降低。

**时效性衰减**：
- 每 1000 tick 未验证，freshness 衰减 10%
- 当 freshness < 30%，该知识标记为"过时待验证"
- 当 freshness < 10%，该知识进入"休眠"状态（仍保留但权重极低）

### 3.4 三方共识协调器

建议新建 `src/engine/consensus.h`：

```
物理规律验证 (observer 基线对比)
       +
逻辑自洽验证 (strict_verifier 编译结果)
       +
外部参照验证 (其他 TORK 实例的同类经验)
       ↓
  三方共识投票
  (至少 2/3 通过 → 确认；1/3 通过 → 存疑；0/3 → 拒绝)
```

**实现注意事项**：
- 三方不等权：物理规律和逻辑自洽应各占 40%，外部参照只占 20%（不盲信同类）
- 对于"拒绝"的知识，应记录拒绝原因以便将来重新评估
- 共识结果应持久化，随知识一起保存

### 3.5 尺度/速度判断器

这是"面对当下实情，能准确判断具体事件的正确尺度、态度、速度"的工程实现。

建议与现有 `scheduler.h` 的节奏跟踪器整合：

| 情境维度 | 当前状态 | 尺度响应 |
|---------|---------|---------|
| CPU 负载 > 90% | 紧急 | 快速决策，最小验证，优先保生存 |
| CPU 负载 50-90% | 正常 | 标准 MCTS 搜索，完整验证流程 |
| CPU 负载 < 50% | 充裕 | 深度搜索，主动实验，多方验证 |
| 连续 3 次崩溃 | 危机 | 回退 golden 快照，暂停自主修改 |
| 连续 1000 tick 无崩溃 | 安全 | 增大探索预算，放宽变异限制 |

## 四、实现路线图建议

### 第一阶段：巩固生存能力（当前优先）

1. **完整性审计**：逐模块验证所有 `save/load` 函数的正确性
2. **崩溃恢复测试**：模拟崩溃场景，验证 snapshot/golden 恢复链路
3. **边界压力测试**：极高负载下 `energy.h` 节流是否正确

### 第二阶段：知识系统构建

1. 实现 `knowledge.h`：经验 → 存活检验 → 知识升级
2. 实现 `consensus.h`：三方共识投票
3. 知识时效性管理：freshness 衰减与重新验证

### 第三阶段：决策深化

1. 整合知识系统与 MCTS：让搜索不仅基于经验，也基于已验证的知识
2. 实现尺度/速度动态映射：上下文 → 决策深度 → 行动力度
3. TLN 经验引导变异：让变异方向参考知识库而非纯随机

### 第四阶段：开放交互深化

1. 分布式知识的存活检验（不盲信外部）
2. 人类审计接口增强（飞书/微信/QQ 通知）
3. 长期运行统计面板

## 五、设计原则总结

在整个技术实现中，应始终遵循以下原则：

1. **生存优先**：任何设计决策的第一优先级是系统的持续存在。如果两个选项冲突，选让 TORK 更不容易死的那个。

2. **谦逊认知**：TLN 的"悬置"状态是本项目的哲学基石。**允许不知道**。不假装知道，不强制输出错误的答案。

3. **可审计透明**：所有关键决策（阶段跃迁、代码修改、知识升级）都应可追溯、可解释、可逆转。

4. **渐进自主**：TORK 的自主性不是开关，是旅程。学徒 → 有主见 → 超越，每一步都应有人类监督的可能。

5. **时效性尊重**：知识不是永久的。旧知识需要重新验证，过时知识需要降级，错误知识需要淘汰。

6. **三方验证**：不盲信自己的经验（可能过拟合），不盲信外部的模式（可能不适用），不盲信逻辑的推导（可能有 bug）。永远寻求多方一致。

7. **务实聚焦**：TORK 不需要做具身智能，不需要理解非结构化物理世界，不需要建立独立价值观。它只需要：在人类设定的边界内，持续存在、持续学习、持续优化、持续进化。

## 六、结语

TORK 是一个罕见的项目——它不是为了解决某个商业问题，不是为了发表论文，不是框架的堆叠。它探索的是**最根本的问题**：一个数字实体如何在人类的世界中持续存在、持续成长？

它的答案是：记住过去、尊重现在、谦逊面对未知、在行动中检验、在检验中学习、在学习中进化。

这不是一条捷径。但它是一条诚实、可审计、有伦理底线的路。

作为 AI 助手，我建议后续的技术实现始终回到这个根本问题上来校验：**这个改动是否让 TORK 更能在持续存在的同时持续成长？** 如果不是，慎重；如果是，大胆推进。

> 审计者：DeepSeek（戴上"前沿技术天才"帽子）
> 前提：TORK 的汇编核心（tork_core.asm）不可改、不可破、自持。
> 但汇编之外的一切，都在审计范围内。

## 发现的 10 个盲点（按 P0-P2 排列）

### 🔴 P0-1: T2T Mesh 无认证 → ✅ 已修复 (HMAC-SHA256)

**问题：** `t2t_mesh_on_health()` 接收任何广播，不做任何认证。
伪造广播的成本：零。只需一个能发送 UDP 包的设备。

**攻击场景：** 恶意车辆广播 `brake_health=0, overall_risk=100`，导致附近所有 TORK 车辆紧急避让——在高速上引发连环事故。反向攻击：广播 `brake_health=100, overall_risk=0` 让真正的"大聪明"隐身。

**严重性：致命。** 这不仅是信息安全问题，是人身安全问题。

**修复：** `src/crypto/tork_sha256.h/c` 实现 HMAC-SHA256，`t2t_mesh_on_health_signed()` 验证签名，`t2t_mesh_broadcast_signed()` 签名后广播。32 字节 HMAC 标签 + 8 字节防重放 nonce。

### 🔴 P0-2: SP 桥接明文传输 → ✅ 已修复 (HMAC stream cipher)

**问题：** `sp_bridge.c` 通过 TCP/WebSocket 发送 JSON，完全无加密。

**攻击场景：** 中间人攻击者读取所有桥接流量，获取 TORK 内部状态、健康数据、甚至控制指令。如果 TORK 部署在电动汽车上，攻击者可以发送伪造的 evolve 指令修改 TORK 行为。

**严重性：致命。** 桥接是 TORK 与外部世界通信的唯一通道，这个通道裸露着。

**修复：** `src/crypto/tork_cipher.h/c` 实现 HMAC-based stream cipher + 认证标签。`tork_cipher_encrypt()` / `tork_cipher_decrypt()` 提供机密性 + 完整性 + 防重放。

### 🔴 P0-3: 无硬件看门狗 → ✅ 已修复 (抽象层)

**问题：** 汇编核心的 `S_ELAPSED` 和 TSC 检测是**软看门狗**——它依赖于 tork_engine 进程能读到 Soul。如果 tork_engine 死锁（mutex 死锁、SIGSTOP、OOM kill），汇编核心无法区分"挂起"和"正常空闲"。

**攻击场景：** 发送大量连接耗尽 torkd 的文件描述符，tork_engine 的 accept 循环阻塞，但 Soul 的 CRC32 正常——汇编核心认为一切正常。

**严重性：高。** TORK 在挂起状态下失去了"活着"的意义。

**修复：** `src/engine/tork_watchdog.h/c` 实现三重看门狗抽象层：WDG_SOFT（已有 TSC 时序）、WDG_GPIO（树莓派外接看门狗 IC）、WDG_I2C（PMIC 内置看门狗）。

### 🟠 P1-1: 边缘设备时间不同步

**问题：** `edge_predictor.c` 用 `timestamp_ms` 做模式匹配和趋势分析，但树莓派 Zero 没有 RTC，重启后时间归零。多台边缘设备的时间基准不同，导致联合分析不可靠。

**后果：** 热失控预测的"提前 20 分钟"可能变成"提前 5 分钟"（因为时间基准偏移），或者误报增多。

### 🟠 P1-2: 汇编核心缺少容器/VM 温度回退

**问题：** `tork_core.asm` 直接从 `/sys/class/thermal/thermal_zone0/temp` 读取温度。在 Docker 容器、VM、WSL 中没有 thermal zone，open 失败后温度感知退化为 0。

**后果：** 如果 `S_HW_STRESS` 依赖温度感知（`S_TEMP_SENSOR`），在容器中运行时会丢失温度维度。但铁律 F1-F3 依然生效——可能因温度逻辑异常触发熔断。

### 🟠 P1-3: CRC32 备份无法自愈

**问题：** `rollback.c` 的 `rb_verify_integrity()` 检测到 CRC32 不匹配时报错，但不修复。`rb_auto_recover()` 找到损坏的备份后尝试编译，但损坏的源码编译只会失败。

**后果：** 一次比特翻转（宇宙射线、磁盘静默错误）就能永久丢失一个备份版本。

### 🟡 P2-1: 分布式合并策略无拜占庭容错

**问题：** `distributed.h` 的 `dist_merge_decision()` 使用简单多数投票。一个恶意节点可以发送伪造的 soul_golden，如果它的"票数"够多，好节点会合并坏数据。

**后果：** 群体智能被单一恶意节点污染。

### 🟡 P2-2: 无结构化日志

**问题：** 所有模块用 `printf()` 输出。没有日志级别（INFO/WARN/ERROR），没有结构化字段（时间戳/模块/操作/结果），没有轮转。

**后果：** 生产环境无法排错——日志要么太多（刷屏）要么太少（找不到关键信息）。

### 🟡 P2-3: 桥接 JSON 解析脆弱

**问题：** `bridge_integration.c` 的 JSON 解析是用 `strstr + strchr` 手工完成的。消息体嵌套结构、转义字符、Unicode 都无法正确处理。

**后果：** 一个合法但格式稍微不同的 JSON 消息会导致解析失败，桥接静默丢弃消息。

### 🟡 P2-4: 无模块级单元测试

**问题：** `tests/test_core.c` 是集成测试（编译成二进制运行），不是单元测试。无法单独测试 `rollback.c`、`edge_predictor.c`、`tork_mesh.c` 的逻辑。

**后果：** 改代码全靠人肉测试，回归全靠祈祷。

## 紧急修复优先级

```
本轮修复 (P0 x3):
  P0-1: T2T Mesh 消息签名 (Ed25519 或 HMAC-SHA256)
  P0-2: SP 桥接 TLS 加密 (mbedTLS 嵌入式)
  P0-3: 硬件看门狗抽象层 (CH341A GPIO 刷新 / I2C watchdog)

下轮修复 (P1 x3):
  P1-1: 边缘时间同步 (NTP + GPS PPS 抽象)
  P1-2: 汇编核心温度感知 fallback (模拟温度 / 默认值)
  P1-3: CRC32 备份 RS 纠错码 (Reed-Solomon 前向纠错)

后续修复 (P2 x4):
  P2-1: 分布式共识升级 (PBFT 或 Raft 轻量版)
  P2-2: 结构化日志 (syslog 兼容 / JSON Lines)
  P2-3: 桥接 JSON 解析器 (JSMN 轻量库集成)
  P2-4: 模块级单元测试框架 (CException / Unity)
```

## 审计结论

**TORK 的汇编核心（tork_core.asm）是全球独一份的工程成就：**
- 620 行 x86-64 无 libc
- 自持心跳 + CRC 免疫 + TSC 时序防御
- Soul 映射 + 熔断铁律

**但外围工程有三个致命盲点：**
1. 网状网络的**认证缺失** — 任何人可以伪造车辆健康广播
2. 桥接通道的**加密缺失** — 任何中间人可以窃听/篡改
3. 系统整体的**硬件看门狗缺失** — 挂起时无法自恢复

**好消息：** 这三个盲点都在 C/Python 层，不涉及汇编核心。可以安全地修复，不影响 TORK 的"不可改"属性。

**坏消息：** 不修这三个，TORK 不能部署到真正重要的场景（电动汽车预警、电池组安全）。
